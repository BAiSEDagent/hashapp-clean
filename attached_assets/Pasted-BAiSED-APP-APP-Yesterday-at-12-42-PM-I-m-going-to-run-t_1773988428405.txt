High-severity findings still open
High 1 — raw wagmi write path still violates ethskills frontend-ux guidance
In Activity.tsx, the Coinbase/non-delegation permission path still uses raw wagmi contract write flow.

Why this matters
Per ethskills frontend-ux:
onchain buttons must stay disabled until confirmation
raw wagmi often resolves at signing, not final confirmed state
this creates duplicate-click / duplicate-submit risk

Risk
User signs, returns, button looks active again, clicks again.

Fix
Refactor the approval path so the UI lock is tied to full tx confirmation state, not just signature initiation.
At minimum:
keep the button disabled until receipt is confirmed and local state refresh completes
add a cooldown/confirmation lock if necessary

High 2 — no proper wrong-network primary state
ETHSkills QA expects a four-state button flow:

Risk
Wrong-network users can still hit strange failures or misleading states.

Fix
Add explicit wrong-network gating anywhere the user is expected to approve or delegate.
Do not let delegation/approval look available on the wrong chain.
